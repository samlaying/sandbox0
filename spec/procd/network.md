# Procd - 网络隔离设计规范

## 一、设计目标

在 **Procd层面** 实现动态网络隔离，无需与 K8s 交互，支持：
- **IP/CIDR 过滤**：精确控制出站流量目标 IP
- **域名过滤**：支持域名、通配符域名（*.example.com）
- **DNS 欺骗防护**：防火墙独立解析域名，不信任沙箱内 DNS
- **私有 IP 黑名单**：默认阻止访问内网 IP
- **动态配置**：运行时通过 API 修改网络策略
- **协议分离**：TCP 通过代理过滤，UDP/ICMP 直接过滤
- **暂停/恢复持久化**：策略在 pause/resume 后保持有效

│   │   ┌─────────────────────────┐         ┌─────────────────────────┐       │  │

│   │   │  │   DNSResolver)    │  │         │                         │       │  │

## 三、网络策略结构

### 3.1 策略模式

| 模式 | 说明 |
|------|------|
| `allow-all` | 允许所有出站流量（不启用防火墙） |
| `deny-all` | 拒绝所有出站流量（默认拒绝） |
| `whitelist` | 白名单模式（仅允许明确指定的流量） |

## 四、nftables 配置

### 4.1 防火墙规则结构

Procd 启动时配置 nftables，规则优先级如下：

1. **Proxy bypass** (最高优先级): `meta mark & 0x2 == 0x2 accept`
2. **预定义黑名单** (私有 IP): `ip daddr @predef_deny drop`
3. **用户拒绝集合**: `ip daddr @user_deny drop`
4. **白名单模式**: TCP 重定向到 TCPProxy

### 4.2 私有 IP 黑名单

默认阻止以下 CIDR：

| CIDR | 说明 |
|------|------|
| `10.0.0.0/8` | 私有网络 A 类 |
| `127.0.0.0/8` | 回环地址 |
| `169.254.0.0/16` | 链路本地 |
| `172.16.0.0/12` | 私有网络 B 类 |
| `192.168.0.0/16` | 私有网络 C 类 |

## 五、TCP 代理（域名过滤）

### 5.1 工作原理

```
┌─────────────────────────────────────────────────────────────────┐
│                    TCP Proxy 域名过滤流程                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  用户进程出站 TCP 连接                                           │
│        │                                                        │
│        ▼                                                        │
│  nftables 重定向到 127.0.0.1:1080                                │
│        │                                                        │
│        ▼                                                        │
│  TCPProxy (SOCKS5)                                              │
│        │                                                        │
│        ├─► 解析目标地址 (IP 或域名)                              │
│        │                                                        │
│        ├─► 如果是域名：                                          │
│        │      - 代理自己 DNS 解析（防欺骗）                      │
│        │      - 检查域名白名单                                   │
│        │                                                        │
│        ├─► 检查 IP 白名单                                        │
│        │                                                        │
│        └─► 建立到目标服务器的连接并双向转发                       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 5.2 域名匹配

- **精确匹配**: `example.com`
- **通配符匹配**: `*.example.com`
- **允许所有**: `*`

## 六、安全与性能

### 6.1 特权容器要求

```yaml
securityContext:
  capabilities:
    add:
      - NET_ADMIN    # Configure nftables firewall rules
```

### 6.2 推荐使用 Kata 运行时

```yaml
apiVersion: v1
kind: Pod
spec:
  runtimeClassName: kata  # 使用 Kata 运行时
```

**Kata 的优势**：
- 特权容器限制在轻量级 VM 内
- 容器逃逸只影响 VM，不影响主机
- 网络策略更新速度相同 (~10ms)

### 6.3 冷启动性能对比

| 方案 | 冷启动延迟 | 安全性 |
|------|-----------|--------|
| K8s NetworkPolicy | ~100-200ms | 高 |
| Procd + 特权容器 | ~10-20ms | 低 |
| **Procd + Kata** | **~10-20ms** | **高** |

| 功能 | E2B | Sandbox0 (Procd层面) |
|------|-----|---------------------|
| **IP/CIDR过滤** | ✅ nftables | ✅ nftables |

| **DNS欺骗防护** | ✅ 独立DNS解析 | ✅ 独立DNS解析 |
| **私有IP黑名单** | ✅ 预定义拒绝集合 | ✅ 预定义拒绝集合 |
| **动态配置** | ✅ 运行时修改 | ✅ HTTP API |

## 八、优势总结

1. **零冷启动延迟**：网络策略在 Procd 层面配置，Pod 认领时无需与 K8s 交互
2. **动态配置**：通过 HTTP API 随时修改网络策略
3. **完整功能**：实现 E2B 所有网络隔离功能
4. **简单部署**：无需复杂的网络 slot 池管理
5. **K8s 原生**：与 ReplicaSet + Idle Pool 完美配合
6. **Kata 友好**：配合 Kata 运行时实现高安全性
